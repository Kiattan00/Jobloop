# 实施计划：JobLoop P0 多版本简历匹配与投递决策工作台

**分支**：`当前工作区` | **日期**：2026-07-02 | **规格**：[spec.md](./spec.md)

**输入**：来自 `specs/001-jobloop-p0/spec.md` 的功能规格

**说明**：本计划基于宪章 2.0.0 与新版 P0 规格生成，目标是为后续 `/speckit-tasks` 提供清晰设计输入。

## 概述

本次 P0 将 JobLoop 收敛为“多版本简历匹配与投递决策工作台”。用户先录入原始简历，生成并确认 2-3 个基础简历版本；随后批量粘贴多个 JD，系统为每个岗位推荐简历版本、判断是否值得投、判断是否需要微调，并提供单岗位完整分析、打招呼话术、面试准备建议与岗位微调建议。

实施路径按用户故事递进：先完成简历库与基础简历生成，再完成批量 JD 输入与岗位分析总览，最后补齐单岗位详情、公司补充信息占位和岗位微调能力。视觉风格以 `docs/JOBLOOP_VISUAL_STYLE.md` 为准，旧过程稿只作为风格参考。

## 技术上下文

**语言/版本**：TypeScript 5.x、React 19、Next.js 16 App Router

**主要依赖**：Next.js、shadcn/ui、`lucide-react`、Biome、TypeScript

**存储**：P0 采用轻量持久化边界，先保证刷新后可恢复原始简历、简历版本、JD、分析结果与微调内容；具体落地可从本地持久化开始，并保留后续替换为数据库的领域模型边界

**测试策略**：默认不新增自动化测试；通过人工核心路径验证、HTTP 页面检查、必要的浏览器烟测，以及 `npm run quality`

**目标平台**：现代桌面浏览器优先，移动端保持基本可访问与不破版

**项目类型**：单体 Web 应用

**性能目标**：核心页面首次可读内容应在 3 秒内出现；批量分析提交后必须展示明确生成中状态；总览页能在 10 秒内支持用户比较至少 3 个岗位的关键判断

**约束**：必须围绕“原始简历 -> 基础简历版本 -> 批量 JD -> 推荐简历版本/是否值得投/是否需要微调 -> 打招呼话术与面试准备建议”的核心路径；不得引入自动投递、真实联网搜索或完整求职 CRM；交付前需通过 `npm run quality`

**规模/范围**：5 个 P0 页面、9 类数据实体、4 类 AI 输出流程、1 套视觉风格基准、1 组 UI/AI 契约文档
