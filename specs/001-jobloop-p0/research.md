# Phase 0 研究：JobLoop P0 多版本简历匹配与投递决策工作台

## 决策 1：P0 聚焦投递前决策，不恢复旧求职 CRM 范围

**Decision**：P0 只覆盖原始简历、基础简历版本、批量 JD、岗位分析总览、单岗位详情和岗位微调。

**Rationale**：宪章 2.0.0 已明确当前产品价值来自“多版本简历 + 批量岗位决策”，而不是完整求职进展管理。收窄范围能更快验证用户是否需要批量比较岗位、选择简历版本和决定是否微调。

**Alternatives considered**：保留 Dashboard、岗位档案、反馈录入、报告页作为 P0。该方案会把产品重新拉回求职 CRM，增加页面与状态复杂度，推迟核心投递前决策验证。

## 决策 2：联网搜索只做占位，不进入核心链路

**Decision**：P0 提供“公司补充信息，可选”输入框和“联网补充公司信息 Beta / Coming soon”入口，但不实现真实联网搜索。

**Rationale**：公司信息能增强投递判断，但搜索质量、来源可信度、解析稳定性会显著扩大 P0 风险。手动补充信息能保留产品方向，也让 AI 分析有可追溯输入。

**Alternatives considered**：直接接入搜索 API 或浏览器插件。该方案适合 P1，但 P0 会被搜索失败、来源可信度和插件交互拖慢。

## 决策 3：AI 输出采用结构化卡片与分区内容

**Decision**：基础简历版本、岗位分析总览、单岗位完整分析和岗位微调都采用结构化输出，不使用单段聊天式文本。

**Rationale**：用户需要比较和决策，结构化字段能支持总览卡片、详情页、追溯来源和后续保存。单段文本难以横向比较，也不利于生成 tasks 和数据模型。

**Alternatives considered**：让模型返回自然语言报告。该方案实现快，但会削弱岗位总览、推荐简历和微调判断的可读性。

## 决策 4：旧前端过程稿保留视觉气质，不保留页面架构

**Decision**：旧 Dashboard、岗位档案、反馈、报告页面只作为视觉参考；新版 P0 页面按 spec 重设为简历库、基础简历生成、批量 JD 输入、岗位分析总览和单岗位详情。

**Rationale**：旧过程稿的图书馆背景、玻璃面板、黑白灰与青色强调已经被认可，但页面结构对应旧 CRM 方向。拆开视觉与信息架构，能保留审美资产，同时避免旧范围反噬新 P0。

**Alternatives considered**：在旧页面上直接改文案。该方案短期快，但会导致路由、组件命名和用户故事继续围绕旧流程。

## 决策 5：存储边界先服务刷新恢复，后续可替换为数据库

**Decision**：P0 计划中先定义稳定领域实体与持久化边界，要求刷新后可恢复；具体实现可以从轻量本地持久化开始，并保留后续替换为数据库的空间。

**Rationale**：当前规格要求数据不丢失，但没有要求多人协作、账号体系或服务端数据库。先保证数据模型稳定，比提前引入复杂数据层更符合宪章的简单实现原则。

**Alternatives considered**：立即设计完整后端数据库和服务层。该方案更接近长期产品，但会扩大 P0 实现面。

## 决策 6：AI 生成能力先通过内部生成边界承载

**Decision**：计划阶段定义“基础简历生成、批量 JD 分析、单岗位完整分析、岗位微调”四类生成契约，具体模型和供应商不在 spec 中绑定。

**Rationale**：用户价值来自输出结构和可追溯链路，模型供应商属于实现细节。通过契约约束输入输出，可以先完成 UI、数据和验证路径，再替换真实模型调用。

**Alternatives considered**：在 plan 中指定某个模型或 SDK。该方案会提前绑定实现依赖，不利于 P0 快速迭代。
