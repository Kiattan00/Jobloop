# JobLoop 宪章

## Core Principles

### I. 简历版本库优先
所有需求、方案和实现都必须优先服务当前 P0 主链路：录入原始简历、生成 2-3 个面向不
同岗位方向的基础简历版本、由用户确认并沉淀到简历库。任何不直接强化这条链路的能力，
都必须默认延后，除非规格文档明确说明它对当前 MVP 的必要性。理由：当前产品价值的
起点不是岗位档案，而是可复用、可选择的多版本简历资产。

### II. 批量岗位决策优先
简历版本库之后的所有功能都必须服务“批量粘贴 JD -> 判断是否值得投 -> 推荐使用哪版简
历 -> 判断是否需要微调 -> 生成打招呼话术与面试准备建议”的投递前决策闭环。若某项能
力不能直接帮助用户更快做出投递选择、简历选择或微调选择，则不得进入 P0。理由：项目
已从求职过程管理收缩为投递前决策工作台，核心价值在于减少筛岗与投递判断成本。

### III. 推荐可追溯与用户控制
每一条 AI 结论都必须能追溯到原始简历、基础简历版本、岗位 JD、可选公司补充信息或
已有岗位分析结果之一；系统必须明确输出推荐简历版本、值得投判断、微调必要性与建议
动作。系统不得静默覆盖用户已确认保存的简历版本，也不得替用户自动投递。理由：投递
决策类产品的可信度来自可解释推荐与用户最终控制权，而非自动替用户做决定。

### IV. 既有技术栈与简单实现
实现必须基于 Next.js App Router、TypeScript 和 npm 工作流；静态质量检查以
Biome 与 TypeScript 为准。UI 优先复用现有 shadcn/ui 组件、主题 token 与
`lucide-react` 图标，避免引入与当前栈重叠的新 UI 体系。理由：固定技术边界能缩短
交付路径，减少 MVP 阶段的维护成本。代码必须围绕当前需求做最短实现，不为未来假设
场景预埋复杂抽象，不因“可能出错”而堆叠防御性分支，不进行与当前目标无关的重构。
仅当抽取能直接降低当前实现复杂度时，才允许做有限复用。

### V. 静态质量闸门与中文文档
所有项目文档、规格、计划、任务和交接材料必须使用中文编写。功能完成的最低交付标准
是：核心路径经过人工验证，且仓库通过 `npm run quality` 或等效静态检查；默认不要求
补测试、不考虑无障碍，也不追加超出需求的健壮性工程，除非功能规格显式提出。理由：
当前阶段以高信噪比交付为目标，质量门槛应聚焦于投递前决策主路径与静态正确性。

## 产品边界

JobLoop 的 P0 页面限定为简历库、基础简历生成页、批量 JD 输入页、岗位分析总览页与
单岗位分析详情页。P0 的 AI 能力限定为：基于原始简历生成 2-3 个基础简历版本；基于
多个 JD 与可选公司补充信息生成批量岗位分析；针对单岗位生成推荐简历版本、值得投判
断、微调建议、打招呼话术与面试准备建议；必要时生成岗位微调版简历或微调正文建议。
以下事项默认不在 P0 范围内：完整求职 CRM 状态流转、面试后反馈录入与复盘闭环、招
聘网站自动抓取、批量自动投递、复杂 Agent、复杂 Word/PDF 排版导出、自动覆盖用户已
确认保存的简历版本。任何计划超出这些边界，必须在规格与计划中明确标
注为“偏离宪章”，并给出业务必要性。

## 开发流程

每个功能必须先形成中文 spec，再形成中文 plan 与 tasks，并在 plan 中逐条完成宪章检
查。任务拆分必须围绕独立用户故事组织，优先保证 P1 用户故事可以单独完成“原始简历
录入与基础版本生成”，并让后续 P2、P3 逐步扩展到批量 JD 决策与单岗位详情。实现过
程中应避免提交无关重构和生成物，文档、代码与任务描述都要使用真实路径和明确范围。
交付说明中必须写明人工验证的核心路径，以及是否已执行 `npm run quality`。

## Governance

本宪章优先于仓库中的其他流程性说明；若出现冲突，以本宪章为准，并在后续同步修正文
档。宪章修订采用语义化版本：

- MAJOR：删除原则、重新定义原则，或引入与现有交付方式不兼容的治理变更。
- MINOR：新增原则、边界或流程要求，且对团队约束有实质扩展。
- PATCH：措辞澄清、格式修复、非语义性补充。

所有 spec、plan、tasks 与交付说明在落盘前都必须检查是否满足本宪章。若存在偏离，
必须在对应文档中显式记录原因、收益与简化方案为何不可行。宪章修订需在同一次变更中
同步更新相关模板与运行文档，并将 `LAST_AMENDED_DATE` 更新为实际修订日期。
